### 컴파일과 인터프리터의 차이는 무엇인가요?
컴파일은 전체 코드를 한 번에 기계어로 변환해 실행 파일을 만들고, 인터프리터는 한 줄씩 읽으며 바로 실행됩니다. <br>

---
### CPU와 메모리 간 병목을 줄이는 기술은 무엇인가요?
CPU와 메모리 간 속도 차이를 줄이기 위해 캐시 메모리를 사용합니다. <br>
캐시는 자주 사용하는 데이터를 저장해 CPU가 빠르게 접근할 수 있도록 도와줍니다. <br>
또한, 파이프라이닝과 멀티코어 구조도 병목 현상을 완화하는 데 기여합니다. <br>

---
### 동기와 비동기의 차이는 무엇인가요?
동기는 한 작업이 끝나야 다음 작업이 실행되고, 비동기는 기다리지 않고 다음 작업을 병렬로 진행됩니다. <br>

---
### API란 무엇인가요?
API는 프로그램 간의 상호작용을 가능하게 하는 인터페이스이빈다. <br>
개발자기 특정 기능을 직접 구현하지 않고도, 다른 프로그램이나 시스템이 제공하는 기능을 호출해 사용할 수 있게 해줍니다. <br>

심화 : <br>
햔데 웹 개발에는 대부분 REST API 형태로 사용됩니다. <br>
클라이언트는 HTTP 요청을 통해 서버의 자원에 접근하고, 서버는 JSON 형태의 응답을 반환합니다. <br>
API는 서로 다른 시스템 간의 의사소통을 규격화하여 확정성과 재사용성을 높여줍니다. <br>
백엔드 개발에서는 API를 설계하는 과정이 곧 시스템 간 계약을 정의하는 작업이기 때문에 매우 중요합니다.
---
### REST API란 무엇인가요?
REST API는 HTTP 프로토콜을 기반으로 클라이언트와 서버 간에 데이터를 주고받는 아키텍처 스타일입니다. <br>
자원은 URI로 식별되고, HTTP 메서드(GET, POST, PUT, DELETE 등)를 사용해 자원을 조작합니다. <br>
REST는 무상태성(stateless)을 유지하며, 클라이언트와 서버 간의 독립성을 보장합니다. <br>

---
### 라이브러리와 프레임워크의 차이는 무엇인가요?
라이브러리는 개발자가 필요할 때 호출해서 사용하는 코드의 모음이고, 프레임워크는 프로그램의 전체 구조와 흐름을 미리 정해놓은 틀입니다. <br>
즉 라이브러리는 개발자가 제어권을 가지고 있고, 프레임워크는 제어권을 개발자가 아닌 시스템이 가집니다.

심화 : <br>
이 차이를 '제어의 역전 IoC' 이라고 합니다. <br>
스프링 프레임워크가 대표적인 예로, 개발자가 직접 객체를 생성하고 연결하는 대신 스프링이 이 과정을 대신 관리합니다. <br>
덕분에 코드의 결합도가 낮아지고 유지보수가 쉬워집니다. <br>
---
### 객체지향 프로그래밍(OOP)의 핵심 원칙은 무엇인가요?
객체지향 프로그래밍은 프로그램을 데이터와 기능이 결합돤 객체들의 상호작용으로 구성하는 방식입니다. <br>
핵심 원칙은 캡슐화, 상속, 다형성, 추상화로 요약됩니다.

심화 : <br>
객체지향의 본질은 '객체 간의 협력' 입니다. 하나의 거대한 프로그램을 여러 객체로 나누어 각각이 역할을 맡고, 메시지를 주고받으며 동작하도록 설계하면 코드의 재사용성과 유자보수성이 크게 향상됩니다. <br>
---
### 백엔드 개발자에게 가장 중요한 역량은 무엇인가요?
백엔드 개발자는 단순히 코드를 작성하는 것 보다, 요청이 들어와 응답이 나갈 때까지의 전체 흐름을 이해하고 설계할 수 있어야 합니다. <br>
이를 위해 HTTP, 네트워크, 데이터베이스, API 설계 원리를 폭 넓게 알아야 합니다.

심화 : <br>
실무에서는 클라이언트의 요청이 컨트롤러로 들어오고, 서비스 로직을 거쳐 데이터베이스에 접근한 뒤, 다시 응답을 생성해 반환하는 구조로 동작합니다. <br>
이 과정에서 성능, 보안, 트랜잭션 처리, 예외 처리 같은 부분을 균형 있게 고려해야 합니다.
---
### 컨텍스트 스위칭이란 무엇인가요?
컨텍스트 스위칭은 CPU가 실행 중인 프로세스나 스레드를 중단하고, 다른 작업으로 전활할 때 이전 상태를 저장하고 새로운 상태를 복원하는 과정입니다. 

심화 : <br>
운영체제는 동시에 여러 프로그램을 실행해야 하므로 CPU를 여러 프로세스가 번갈아 사용합니다. <br> 
이때 실행 중이던 프로세스의 레지스터, 프로그램 카운터 등의 정보를 PCB에 저장하고, 새 프로새스의 정보를 복원해 실행을 이어갑니다. <br>
이 과정은 필수적이지만 비용이 발생하므로, 너무 잦은 스위칭은 오히려 성능 저하로 이어질 수 있습니다.

---
### 프로그램이 실행될 때 전체 흐름을 요약해 설명해주세요.
사용자가 프로그램을 실행하면, 운영체제가 저장장치에 있는 코드를 메모리로 올리고 프로세스를 생성합니다. <br>
CPU는 메모리에 올라온 명령어를 하나씩 읽어 해석하고 실행하며, 결과를 다시 메모리나 입출력 장치를 통해 전달합니다.

심화 : <br>
이 과정은 CPU의 명령어 사이클, 메모리의 계층 구조, 프로세스 스케줄링, 스레드 동작 등이 유기적으로 맞물려 돌아가면서 이루어집니다.<br>

---
### 사용자가 브라우저 주소창에 google.com을 입력하면 어떤 일이 발생하나요?
사용자가 브라우저 주소창에 google을 입력하고 엔터를 누르는 순간, 브라우저는 먼저 도메인 이름을 IP 주소로 변환하기 위한 DNS 요청을 보냅니다 <br>
DNS 서버는 google,com에 해당하는 실제 서버의 IP 주소를 찾아 브라우저에 반환합니다.  <br>
이 IP 주소를 통해 브라우저는 구글 서버와 연결을 시도하게 됩니다.

심화 : <br>
DNS 요청은 일반적으로 로컬 캐시 ➝ 운영체제 ➝ DNS 서버 ➝ 루트 DNS 서버 ➝ TLD 서버 ➝ 권한 DNS 서버 순으로 진행됩니다. <br>
만약 이전에 방문한 적이 있어 캐시에 IP가 저장되어 있다면 DNS 질의 과정을 생략합니다. <br>
IP를 획득한 후 브라우저는 TCP 3-way handshake를 통해 서버와 연결을 설정하고, 이후 HTTPS 프로토콜을 통해 보안 통신을 시작합니다.

---
### TCP 3-way Handshake는 어떤 과정을 거치나요?
TCP 연결은 신뢰성 있는 통신을 위해 3-way Handshake 절차를 거칩니다. <br>
먼저 클라이언트가 SYN 패킷을 보내고, 서버는 SYN-ACK로 응답한 뒤, 클라이언트가 다시 ACK를 보내면 연결이 성립됩니다. <br>

심화 : <br>
이 과정을 통해 클라이언트와 서버는 서로가 통신할 준비가 되어 있음을 확인하고, 데이터 송수신이 안전하게 이루어질 수 있는 상태를 만듭니다. <br>
이후 HTTPS의 경우 SSL/TLS Handshake가 이어지며ㅡ 암호화된 세션 키를 교환해 안전한 데이터 통신 환경을 구축합니다.

---
### 연결이 완료된 후, 브라우저는 어떤 요청을 서버로 보내나요?
연결이 완려되면 브라우저는 HTTP 요청을 보냅니다 <br>
예를 들어 구글 홈페이지를 요청한다면 "GET / HTTP/1.1"과 같은 요청 라인과 함께 호스트, 사용자 에이전트, 쿠키 등 다양한 헤더 정보를 포함시킵니다. <br>

심화 : <br>
이 요청은 TCP 세그먼트로 분할되어 서버로 전송되며, 서버는 이를 받아 HTTP 요청을 처리합니다. <br>
서버는 요청된 자원을 찾아 HTTP 응답을 생성하고, 상태 코드, 헤더, 본문 데이터를 포함해 클라이언트로 반환합니다.  

---
### 서버에서는 어떤 일이 일어나나요?
서버는 클라이언트로부터 받은 HTTP 요청을 분석하고, 요청된 자원에 따라 적절한 처리를 수행합니다. <br>
예를 들어 데이터베이스에서 정보를 조회하거나, 파일 시스템에서 정적 파일을 읽어오는 작업이 이루어집니다. <br>
처리가 완료되면 서버는 HTTP 응답을 생성해 클라이언트로 반환합니다. <br>

심화 : <br>
서버는 요청을 처리하는 동안 미들웨어, 라우팅, 컨트롤러, 서비스 레이어 등 다양한 컴포넌트가 협력합니다. <br>
이 과정에서 인증, 권한 검사, 로깅, 캐싱 같은 부가 기능도 함께 수행됩니다. <br>
